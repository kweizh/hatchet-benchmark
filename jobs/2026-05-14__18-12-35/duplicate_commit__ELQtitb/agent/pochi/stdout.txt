# Duplicate a Commit in Jujutsu ## Background Jujutsu (`jj`) is a modern, Git-compatible version control system. Each commit is identified by both a *change ID* (a stable identifier that survives rewrites) and a *commit ID* (the underlying Git SHA). The `jj duplicate` command creates a copy of an existing commit as a sibling — the new commit keeps the same description and file changes, but is assigned a brand-new change ID. A Jujutsu repository is already initialized at `/home/user/myrepo`. The repo is colocated with Git (it has both a `.jj/` and a `.git/` directory) and user identity has been pre-configured. There is exactly one user-created commit with the description `Feature A` that introduces a file named `feature_a.txt` containing the text `feature A content`. The working copy `@` is a new empty commit on top of that commit. ## Requirements * Locate the change ID of the existing commit whose description contains the substring `Feature A`. * Use `jj duplicate` to create a duplicate of that commit. The duplicate must: * Have the same description (`Feature A`). * Contain the same file `feature_a.txt` with the same content `feature A content`. * Have a *different* change ID from the original (i.e., it must be a brand-new sibling commit, not an alias of the original). * After the operation, the repository must contain at least two distinct commits whose description matches `Feature A`. ## Implementation Guide 1. `cd /home/user/myrepo` 2. Find the change ID of the existing `Feature A` commit using the `description(substring:...)` revset, for example: ```bash jj log -r 'description(substring:"Feature A")' --no-graph -T 'change_id ++ "\n"' ``` (Note: in jj 0.38, the plain `description("Feature A")` form does an exact match including any trailing newline; use the `substring:` form to match the description text.) 3. Run `jj duplicate <change_id>` with that change ID. This creates a new sibling commit with the same description and changes but a new change ID. 4. Confirm the duplication by re-running the log command and verifying that two lines (two distinct change IDs) are printed. ## Constraints * Project path: /home/user/myrepo * Use the real `jj` binary that is pre-installed in the environment. Do not mock or stub `jj`. * Do not modify the original `Feature A` commit's description or its file contents. * Do not introduce additional commits with the description `Feature A` by any means other than `jj duplicate` (e.g., do not hand-create another commit with the same description).
